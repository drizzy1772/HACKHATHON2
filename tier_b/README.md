# Tier-B «CTBR Pirouette»: gym-pybullet-drones + PPO/SAC

Прямий наступник Tier-A+ («пірует, який заборонила фізика», див.
`env_attitude/README.md`). Там було виміряно: за фіксованої hover-тяги крен у
прямому польоті лише зносить дрон убік і садить по висоті — knife-edge вимагає
`T = mg/cos φ` (1.5× на 47°, 3.9× на 75°). Тут тяга стає четвертим каналом дії:
**CTBR** `a = [c, ω_x, ω_y, ω_z]`, `c ∈ [0, 2g]` — і маневр стає фізично
можливим. Два алгоритми (PPO і SAC, stable-baselines3) тренуються на одному
середовищі з однаковими сідами лісів і порівнюються чесно.

## Швидкий старт

```bash
# середовище (окремий conda env — torch/gymnasium НЕ потрапляють у ядро кіта)
conda create -y -n drones python=3.10
git clone https://github.com/utiasDSL/gym-pybullet-drones.git ~/Desktop/gym-pybullet-drones
conda run -n drones pip install -e ~/Desktop/gym-pybullet-drones tensorboard pyyaml 'setuptools<81'
# macOS: pybullet збирається з сорсів; на нових SDK потрібен обхід
#   CFLAGS="-Dfdopen=fdopen" pip install pybullet
# (vendored zlib у pybullet визначає fdopen→NULL під TARGET_OS_MAC — ламає _stdio.h)

conda run -n drones python -m pytest tier_b/admin/tests   # 67 тестів
conda run -n drones python tier_b/train_ppo.py      # 2M кроків, ~1 год на M4
conda run -n drones python tier_b/train_sac.py      # 2M кроків, довше
conda run -n drones python tier_b/compare.py        # PPO vs SAC → tier_b/admin/reports/
conda run -n drones python tier_b/viz_lidar.py --model tier_b/runs/ppo/seed0/final_model.zip
```

Головний venv репозиторію лишається чистим: усі pybullet/torch-тести
скіпаються без conda env (`pytest.importorskip`), кореневий `pytest` зелений.

| `envs/forest_gen.py`, `envs/trajectory.py` | seeded ворота + вейпоінти + потенціали |
| `rewards.py` | диспетчер reward-ів: референс у `admin/rewards_solution.py`, учнівська реалізація у `scaffold/student_rewards.py`, `Φ(terminal)=0` |
| `callbacks.py`, `train_common.py`, `train_ppo.py`, `train_sac.py` | тренування |
| `eval.py`, `compare.py` | held-out оцінка (сіди 900000+) і порівняння |

**Частоти**: фізика 240 Гц, політика 48 Гц (чесне відхилення від «50 Гц» у ТЗ:
240/50 не ціле; 48 = 240/5). **CTBR**: у бібліотеці такого ActionType немає —
підклас передає батькові `act=RPM` і повністю перевизначає
`_actionSpace`/`_preprocessAction`: rate-P `τ = (J/τ_c)(ω_sp − ω)` → міксер
CF2X (знакові рядки зняті з інстальованого сорсу `BaseAviary._dynamics` і
підтверджені емпірично) → сатурація «тяга-пріоритет» → RPM. Валідовано
відкритим контуром: c=g зависає, c=2g дає чистий +g, трекінг ±6 рад/с у межах
20% на 48 Гц — substep-фолбек не знадобився.

**LIDAR (34 = 8+6+4+16)**: повне коло XOY (8×45°); YOZ додає 6 (±y уже є);
XOZ додає 4 (±x, ±z уже є); 4 «бісектрисні» великі кола (площини x=±y і y=±z)
дають усі 8 напрямків (±½,±½,±√2/2) і всі 8 (±√2/2,±½,±½). Свідома
анізотропія — покриття зміщене до осі польоту. Чесне обмеження: крок 45°
надійно «бачить» стовбур r=0.2 м лише з ~0.5–1 м; глобальну навігацію несе
вейпоінт-спостереження, LIDAR — останній метр. Виміряний нюанс pybullet:
`collisionFilterMask` променя фільтрує лише тіла з ЯВНО заданою групою — дрон
і землю треба явно ставити в групу 1, інакше промені влучають у них.

**Колізія**: XY-тінь еліпсоїда — точний еліпс `M = ĀĀᵀ`; тест проти круга
дерева — консервативна support-функція `depth = r_t + √(nᵀMn) − δ`. Чистий
крен відтворює `presented_half_width` з `env_attitude/occupancy.py` тотожно
(крос-чек у тестах); brute-force по 4000 точках поверхні: жодних хибних
«вільно», |Δdepth| < 5 мм.

## Курикулум

Щілина `G` між ПОВЕРХНЯМИ стовбурів (r=0.2 м), потрібний крен із
`w(φ) = √((0.12cosφ)² + (0.04sinφ)²)`, запас 1 см:

| рівень | G, м | потрібний крен | режим |
|---|---|---|---|
| L0–L2 | 1.20 / 0.80 / 0.55 | 0° | навчання польоту й прицілювання |
| L3 | 0.40 | 0° (запас 7 см) | рівний, легкий |
| L4 | 0.30 | 0° (запас 2 см) | рівний, тісний |
| L5 | 0.26 | ≥ ~25° | перший крен |
| L6 | 0.20 | ≥ ~45° | аналог Tier-A+ «angled» |
| L7 | 0.16 | ≥ ~60° | крутий |
| L8 | 0.12 | ≥ ~75° | майже knife-edge (корпус 0.24 м!) |

Перехід: success rate ≥ 0.7 за останні 200 епізодів поточного рівня.

## Що зламалося і як лагодили (виміряно, не вгадано)

Кожна корекція — з даних конкретного прогону; всі три залишені в
`config.yaml` з коментарями.

1. **Cash-out потенціалу робив колізії вигідними** (smoke 50k: collision rate
   РІС 9%→22%). `Φ(terminal)=0` конфіскує потенціал — тобто на краші агент
   «обналічує» до +70 shaping, а ковзна колізія коштувала лише −50.
   → `collision_base −100`, лобовий −200; структурний тест
   `test_collision_dominates_max_potential_cashout`.

2. **Рента за стояння** (probe 300k: епізоди по 532 кроки, 0 воріт,
   mean_r +331). Форма `γΦ′−Φ` з γ=0.99 платить `−(1−γ)Φ = +0.58/крок` просто
   за існування далеко від цілі; velocity-jam не ловить «кружляння» зі
   швидкістю >0.05 м/с. → shaping у телескопічній формі `Φ′−Φ`
   (`shaping_gamma: 1.0`; саме цю форму перевіряє loop-sum-zero тест кіта) +
   progress-stall обрив (немає покращення залишку маршруту на ≥0.1 м за 5 с).
   Якір: `test_standstill_earns_exactly_zero_shaping`.

3. **Драбина починалася з непрохідного для новачка** (probe2 300k: політика
   ЛІТАЄ впевнено, але 0 воріт за 1655 епізодів — щілина 0.40 м вимагає ±8 см
   точності). → три легкі рівні 1.20/0.80/0.55 м на початку драбини.

Також виміряні й виправлені: `std=1` на 4 CTBR-каналах → перекиди і ~60%
епізодів out-of-bounds за 1.6 с (→ `log_std_init: -1`); лічильник подій на
`step_counter` батька, який інкрементується ПІСЛЯ хуків (→ власний
`_ctrl_ticks`).

## Результати

Обидва алгоритми: 2M кроків, сід 0, однакові сіди лісів за конструкцією.
Головний артефакт: **SAC навчився КРЕНИТИСЬ крізь щілину** — у успішних
прольотах L5 (0.26 м) виміряний крен біля воріт **35–39°** (рівний політ
крізь 0.26 м вимагав би ±1 см точності при ширині корпусу 0.24 м). Це саме
той маневр, який Tier-A+ із фіксованою hover-тягою виконати не міг фізично.

**Тренування** (за monitor CSV; повні криві — `tier_b/admin/reports/tier_b_ppo_vs_sac.png`):

| algo | епізодів | макс. рівень | per-level success (успіхи/епізоди) |
|---|---|---|---|
| PPO | 10715 | **L3** (0.40 м) | L0: 1077/4752 · L1: 749/1471 · L2: 1777/3366 · L3: 318/1126 |
| SAC | 8876 | **L5** (0.26 м, крен!) | L0: 259/2993 · L1: 153/200 · L2: 251/409 · L3: 762/2537 · L4: 681/2031 · L5: 170/706 |

**Held-out оцінка** (сіди 900000+, ніколи не бачені у тренуванні; 50
детермінованих епізодів на рівень; успіх = повна траса, 3 воріт + фініш):

| рівень (щілина) | PPO | SAC |
|---|---|---|
| L0 (1.20 м) | 70% | 46% |
| L1 (0.80 м) | 78% | 64% |
| L2 (0.55 м) | 86% | **80%** |
| L3 (0.40 м) | 70% | 62% |
| L4 (0.30 м) | 16% | **36%** |
| L5 (0.26 м, потрібен крен) | **0%** (50/50 колізій) | **16%**, крен у щілині 35–39° |

Читання чесне: PPO стабільніший на рівнях, де достатньо рівного польоту
(L0–L3), SAC глибше просунувся драбиною і єдиний виконує маневр із креном.
`viz_lidar.py --model .../sac/seed0/final_model.zip --level 5` показує це наживо.

**Ніхто не дійшов до knife-edge за 2M**: L6–L8 (0.20/0.16/0.12 м, крен
45–75°) лишилися нескореними — SAC на момент вичерпання бюджету тримав
~24% на L5 (поріг переходу 70%). Це виміряна стеля бюджету, не приховуємо:
наступний крок — довший бюджет або warm-start із чекпоінта L5.

## Чи можна крутіше за 39°? Зонд «флік» (`probe_flick.py`)

Питання після головного результату: SAC пройшов 0.26 м на 35–49° — а більші
кути? Геометрично крутіше = ЛЕГШЕ (профіль тоншає: 0.101 м на 37° → 0.069 на
60° → 0.050 на 75°), але динаміку перевірив скриптований зонд (діагностичний
оракул у стилі Tier-A, 13 ітерацій ручного тюнінгу). Виміряні закони:

1. **Момент крену вимагає тяги.** При c=0.05g мотори майже стоять і
   диференціального моменту немає: фактична швидкість крену ~1.3 рад/с проти
   6 заявлених. «Балістичний флік із нульовою тягою» не може закрутитись —
   крутити треба на c≈1g.
2. **Бічний імпульс крену залежить лише від кінцевих кутів**:
   Δv_y = c(1−cosφ)/ω̄ — шлях профілю не має значення. Виміряно: S-мах
   −60°→+60° на c=1g зніс дрона на +0.77 м (формула дає ~0.8). Жодна
   хитрість профілю крену цього не скасовує — лише зустрічний «краб»-підхід.
3. **Висота — дешевий ресурс, y — дорогий.** Дерева на всю висоту: просадка
   до ~1 м безкоштовна (перед-набір + відновлення на 2g), а допуск по y на
   L6 — ±3.1 см при 60°.

Підсумок зонда: одні ворота на 60.5° крену пройдено (продемонстровано
фізичну можливість), але повна траса 0/10 на L6–L8 у всіх конфігураціях
(45–85°): вікно (приціл, зустрічна v_y, таймінг фаз крабпідхід → S-флік →
coast → відновлення) — сантиметри й десяті м/с у 4-вимірному просторі
параметрів, ручний PD-автомат його не тримає. Це і є виміряна відповідь,
чому тут потрібен RL/оптимізатор траєкторій, а не контролер з констант: SAC
свої 35–49° знайшов сам, і тепер відомо, ЯКУ саме навичку він мусить
відкрити для L6–L8.

## Анти-reward-hacking аналіз

«Зависнути подалі від дерев» строго гірше за проліт: стояння дає рівно 0
shaping (телескопічна форма) мінус step_cost, velocity-jam обриває за 2 с,
progress-stall — за 5 с, і обидва БЕЗ бонуса (truncation, bootstrap
зберігається — дисципліна terminal-vs-truncated кіта). «Розбитися, обналічивши
потенціал» гірше за все: −100…−200 > максимального cash-out ~78 у будь-якій
точці, де є дерево (структурний тест). ω-штраф свідомо малий: сплеск 6 рад/с
на 0.3 с коштує ≈5 ≪ бонус воріт 25 — маневр не задушений. Залишковий ризик:
на верхніх рівнях обережна політика може віддавати перевагу stall-обриву перед
воротами (EV колізії проти EV обриву) — це чесно вимірюється per-level
статистикою, а не приховується.
